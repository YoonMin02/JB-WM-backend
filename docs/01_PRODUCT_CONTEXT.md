# 01 · 제품 컨텍스트

## 한 문장 정의

> JB WM Agent는 고객의 **건강·자산·보험·대출 데이터를 상시 관찰**하다가, 이벤트가 감지되면 **고객의 잠재 의도를 추론**하고, 계획을 세워 리스크를 검토한 뒤, **고객 승인 범위 안에서** 예약·청구·조정 같은 후속 액션을 수행하고, 결과를 검증하여 다음 판단에 반영하는 **능동형 Life Event Agent**입니다.

챗봇이 본질이 아닙니다. 본질은 **이벤트 기반 상태 그래프 + LLM 추론기**이고, 챗은 인터페이스입니다.

## 문제 정의

오늘날 고객은 건강·금융이 얽힌 상황에서 **모든 판단을 직접** 해야 합니다.

예시 상황:
```
- 최근 건강검진에서 혈압 수치 상승
- 수면 데이터 악화
- 최근 의료비 지출 증가
- 포트폴리오 고위험 주식 비중 70%
- 3개월 뒤 대출 상환 예정
- 실손보험은 있지만 심혈관 특약은 없음
```

기존 서비스라면 고객이 직접:
```
1. 건강검진 결과 확인
2. 병원 갈지 판단
3. 보험 보장 범위 찾아보기
4. 의료비가 자산계획에 주는 영향 계산
5. 투자 비중 줄일지 고민
6. 상담사 문의
```

→ 고령층에게 특히 부담이 크고, 각 단계가 서로 다른 시스템·전문성을 요구하며, 놓치면 **보장 공백·현금흐름 위기·부적합 투자**라는 실질적 손실로 이어집니다.

## 대상 사용자

| 사용자 | 역할 |
|---|---|
| **고객 (주 타깃: 고령층)** | 데이터 제공 + 에이전트 제안에 대한 승인/거절. "신경을 끄는" 경험 |
| 어드바이저 (부차) | 고객 상태·에이전트 판단을 모니터링, 필요 시 개입 |
| 운영자 | 규정·통계 데이터·정책 규칙 관리 |
| 코딩 에이전트 | 문서를 따라 구현·유지보수 |

> 고령층 타깃이므로 큰 글씨·쉬운 용어·명확한 승인 UX가 핵심 차별점입니다. UI 언어는 한국어 우선(i18n 구조는 ko/en 대비).

## 핵심 차별점

1. **능동성** — 고객이 요청하지 않아도 시스템이 먼저 변화를 감지하고 판단한다. (기존 챗봇: 물어봐야 동작)
2. **건강 ↔ 금융 연계** — 건강 이벤트를 보험·자산·투자·생애설계 액션으로 연결한다. (단순 포트폴리오 추천이 아님)
3. **의도 = 상태** — 고객의 잠재 의도를 상태 그래프로 구조화해 중복 실행·승인 누락·맥락 꼬임을 코드로 방지한다.
4. **Capability 보안** — 민감 액션 권한을 LLM에 아예 주지 않는다. 신뢰가 아니라 구조로 안전을 보장한다.

## MVP 범위

핵심 1개 플로우를 **수직으로** 완성합니다 (모든 의도를 다 만들지 않음). 자세히는 [11_IMPLEMENTATION_ROADMAP.md](11_IMPLEMENTATION_ROADMAP.md).

### MVP 시나리오 (데모용)

```
건강 이상 신호 감지 (혈압 상승 + 수면 악화)
→ 보험/자산 영향 분석 (보험 공백 + 3개월 현금흐름 리스크)
→ 고객 의도 추론
→ 맞춤 액션 카드 생성
→ 고객 승인 (예: "보험·현금흐름만, 투자는 보류")
→ 병원 예약 mock / 보험 청구 서류 mock / 포트폴리오 조정안 생성
→ 결과 확인 화면
→ 고객 선호 업데이트 (장기 메모리)
```

데모 화면 예시:
```
[Agent 알림]
최근 건강 데이터와 자산 상태를 함께 분석한 결과,
3개월 내 현금흐름 리스크가 상승했습니다.

원인:
- 최근 의료비 지출 증가
- 혈압 관련 추가 검진 가능성
- 3개월 뒤 대출 상환 예정
- 고위험 투자 비중 70%

추천 액션:
1. 가까운 병원 검진 예약
2. 실손보험 보장 범위 확인
3. 3개월 비상자금 확보 플랜 생성
4. 포트폴리오 위험도 낮춘 대안 보기

어떤 것부터 진행할까요?
```

고객이 "보험이랑 현금흐름만 먼저 봐줘. 투자는 아직 건드리지 마"라고 하면, 상태가:
```
InsuranceIntent     → Approved
AssetDefenseIntent  → Approved
InvestmentAdjustIntent → Deferred
HealthCareIntent    → Pending
```
로 바뀝니다. **이것이 "고객 의도가 상태로 표현된다"는 의미입니다.**

## MVP 비목표

- 실제 병원 예약 / 실제 보험 청구 / 실제 금융 거래 (전부 mock)
- 완전 자동 금융 자문 실행 (항상 인간 승인)
- 프로덕션급 컴플라이언스 워크플로우
- 다중 LLM 공급자 마이그레이션 (구조만 대비, 구현은 Codex 1종)
- 실시간 시장 데이터, 벡터 RAG (나중 고도화)

## 제품 원칙

에이전트는 **분석과 제안**을 생성한다. **백엔드(코드)가 상태와 권한을 통제**한다. 민감한 액션은 **고객이 승인**한 뒤에만 실행된다.
